1. Executive Summary

MXSuite is a multi-tenant data onboarding platform designed to replace the current spreadsheet-based data migration process used by GrowthZone MemberSuite. Instead of generating Excel files for customers to manually fill in, MXSuite provides a web-based interface where customers can upload their data files, visually map fields to GrowthZone's canonical model, validate transformations via dry runs, and execute full data loads вҖ” all with complete audit trails and real-time collaboration.

Key Value Propositions:
  • ELT over ETL: Load raw data first, transform in the database with full SQL power, auditability, and repeatability
  • Self-Service Onboarding: AI-assisted field mapping reduces GrowthZone staff involvement
  • Platform Architecture: Extensible, Salesforce-like platform вҖ” not a single-purpose tool
  • Complete Audit Trail: Every action is logged вҖ” who did what, when, and whether it was a customer or GrowthZone staff action
  • Cloud-Agnostic: Docker/Kubernetes deployment runs anywhere

2. Problem Statement

The current GrowthZone onboarding process has several inefficiencies:

  • Spreadsheet-based mapping: GrowthZone generates an Excel spreadsheet template, the customer fills it in, and GrowthZone processes it. This is error-prone, doesn't scale, and pushes transformation burden onto non-technical customers.
  • ETL in-flight transformations: Data is transformed during transit, making it harder to debug, audit, and re-run when issues arise.
  • No audit trail: Limited visibility into who made changes and when.
  • Manual intervention: GrowthZone staff must heavily assist customers through the process.
  • Cost: Current infrastructure uses Snowflake, which is expensive for this use case.

Each plan stores its mapping definition as a JSONB column:

{ "sourceSchema": [ { "name": "member_name", "type": "string", "path": "member_name" }, { "name": "join_date", "type": "string", "path": "join_date" } ], "targetSchema": [ { "name": "fullName", "type": "string", "path": "contact.fullName" }, { "name": "memberSince", "type": "date", "path": "membership.startDate" } ], "mappings": [ { "id": "m1", "sourceField": "member_name", "targetField": "fullName", "transformation": "TRIM | TITLE_CASE" }, { "id": "m2", "sourceField": "join_date", "targetField": "memberSince", "transformation": "DATE_PARSE('MM/dd/yyyy')" } ], "validationRules": [ { "field": "fullName", "rule": "NOT_EMPTY", "message": "Name is required" }, { "field": "memberSince", "rule": "VALID_DATE", "message": "Invalid date format" } ] }

Execution Flow

1. User submits Plan for execution (Dry Run or Full Run) 2. PlanRun record created with status QUEUED 3. Server picks up run, sets status to RUNNING 4. For each record in source data: a. Apply field mappings b. Execute transformation DSL c. Validate against rules d. On Dry Run: record result but don't commit e. On Full Run: write to target database 5. Set status to COMPLETED or FAILED 6. Store result summary and errors as JSONB 7. Broadcast completion event via WebSocket

Logging Pipeline

Structured logging with automatic trace correlation ensures every log line can be linked back to a distributed trace.

Log Pattern (Console + File): %d{ISO8601} [%thread] %-5level %logger{36} traceId=%X{trace_id} spanId=%X{span_id} - %msg%n Profile-based routing: local вҶ’ Console appender + Rolling file (100MB, 30-day retention, 3GB cap) docker/k8s вҶ’ Console appender + OTEL appender (ships to Collector вҶ’ Loki)

Log вҶ’ Trace Correlation

Every log line includes the OpenTelemetry trace_id and span_id. Grafana's Loki datasource is configured with a derived field that turns trace IDs into clickable links to Tempo, enabling instant navigation from a log line to the full distributed trace.

Distributed Tracing

OpenTelemetry auto-instrumentation captures traces across HTTP requests, database calls, and Redis operations. Traces are exported via OTLP to the OTEL Collector, which forwards them to Tempo.

Trace вҶ’ Metric Correlation

Tempo's metrics generator produces span metrics and service graphs, written back to Prometheus via remote write. This enables:

  • Service Map: Visual graph of service dependencies with request rates and error percentages
  • Span Metrics: Auto-generated latency histograms and throughput counters from trace data
  • Exemplars: Direct links from Prometheus metric data points to individual traces

Horizontal Scaling вҖ” Single Pane of Glass

All observability components are designed for multi-instance awareness:

  • Prometheus scrapes all API instances via Docker DNS or Kubernetes service discovery. Every metric includes an instance label.
  • Infrastructure dashboard shows all instances side-by-side: CPU, heap, request rate, thread count, HikariCP pools per instance.
  • Stat panels aggregate across instances using sum() and count(up) for total request rate, healthy instance count, and overall error rate.
  • Loki aggregates logs from all instances with the service_name label, queryable by instance.
  • Tempo captures distributed traces across instances, showing the full request path through load-balanced services.
  • Redis Pub/Sub ensures WebSocket events reach all instances, so the WebSocket connection gauge reflects the true global count.
Kubernetes-Ready: The Prometheus scrape config includes a commented Kubernetes service discovery block. When deployed to K8s, simply enable the kubernetes_sd_configs section вҖ” Prometheus will automatically discover and scrape all pods with the mxsuite-api label.

  • Append-only audit_events table вҖ” never modified or deleted

  • Queryable by tenant, entity, actor, date range, platform-only filter
  • Trace ID in audit events links each audit record back to the full distributed trace in Tempo
  • Custom audit.events Micrometer counter tracks audit event rates on the Plan Execution dashboard
