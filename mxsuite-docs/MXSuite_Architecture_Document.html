1. Executive Summary

MXSuite is a multi-tenant data onboarding platform designed to replace the current spreadsheet-based data migration process used by GrowthZone MemberSuite. Instead of generating Excel files for customers to manually fill in, MXSuite provides a web-based interface where customers can upload their data files, visually map fields to GrowthZone's canonical model, validate transformations via dry runs, and execute full data loads — all with complete audit trails and real-time collaboration.

Key Value Propositions:
  • ELT over ETL: Load raw data first, transform in the database with full SQL power, auditability, and repeatability
  • Self-Service Onboarding: AI-assisted field mapping reduces GrowthZone staff involvement
  • Platform Architecture: Extensible, Salesforce-like platform — not a single-purpose tool
  • Complete Audit Trail: Every action is logged — who did what, when, and whether it was a customer or GrowthZone staff action
  • Cloud-Agnostic: Docker/Kubernetes deployment runs anywhere

2. Problem Statement

The current GrowthZone onboarding process has several inefficiencies:

  • Spreadsheet-based mapping: GrowthZone generates an Excel spreadsheet template, the customer fills it in, and GrowthZone processes it. This is error-prone, doesn't scale, and pushes transformation burden onto non-technical customers.
  • ETL in-flight transformations: Data is transformed during transit, making it harder to debug, audit, and re-run when issues arise.
  • No audit trail: Limited visibility into who made changes and when.
  • Manual intervention: GrowthZone staff must heavily assist customers through the process.
  • Cost: Current infrastructure uses Snowflake, which is expensive for this use case.

6. Multi-Tenancy

Strategy: Shared PostgreSQL schema with tenant_id column discrimination and row-level security (RLS).

Tenant Types

TypeDescriptionCapabilities
PLATFORMMXSuite/GrowthZone — the super-tenantFull control over all customer tenants, cross-tenant workspaces, user management
CUSTOMERGrowthZone customer organizationManages own workspaces, projects, plans. Cannot see other tenants.

Data Isolation

  • Every tenant-scoped query includes a tenant_id filter
  • The TenantContext (ThreadLocal) is set by the JWT authentication filter
  • Platform users can override the tenant context via X-Tenant-Id header
  • PostgreSQL Row-Level Security provides a database-level safety net

Permission Model

Each Workspace and Project has:

  • Owner — creator by default, can be transferred
  • Access List — explicit grants with level: VIEWER EDITOR ADMIN

OIDC / SSO Ready

The architecture is designed for future identity provider integration. The spring-boot-starter-oauth2-resource-server dependency is already included. To add Okta, WSO2, or any OIDC provider:

  • Create a new Spring profile (e.g., okta, wso2)
  • Configure spring.security.oauth2.resourceserver.jwt.issuer-uri
  • The JWT filter auto-switches based on active profile
  • No code changes required — configuration only

User Invitation System

Users are onboarded through an email invitation workflow:

Admin sends invite ──→ Email with token link ──→ User clicks link │ │ │ ▼ │ Accept form (name + password) │ │ │ ▼ │ User account created │ │ └── Can: Resend / Cancel ▼ User can now log in
  • Invitation tokens expire after 7 days
  • TENANT_ADMIN can invite TENANT_USER or TENANT_ADMIN roles
  • PLATFORM_ADMIN/SUPPORT can invite any role
  • Duplicate checks prevent inviting existing users or re-sending pending invites
  • Admins can resend (new token, extended expiry) or cancel invitations

Password Recovery

  • User requests reset via email — always returns success (prevents email enumeration)
  • Reset tokens expire after 1 hour, single-use
  • All reset events are logged with client IP for forensic analysis

Breach-Hardened: GrowthZone has experienced data breaches in previous systems. MXSuite is built with defense-in-depth security at every layer.

Email Notification Service

MXSuite includes an async email service for:

  • User invitations — Welcome email with accept link
  • Password recovery — Reset link with 1-hour expiry
  • Admin alerts — Critical system warnings (database issues, heap/memory problems, instance failures)

Grafana alerting rules (Prometheus) can route critical alerts to the admin email endpoint for database, heap, connection pool, and instance-down events.

8. Plan Engine — ELT Pipeline

Plan Lifecycle

DRAFT ──→ PUBLISHED ──→ ARCHIVED ↑ │ │ ├──→ Dry Run (validate, analyze, no side effects) │ └──→ Full Run (execute transformation and load) │ └── Edit field mappings, transformations, validation rules

Plan Definition (JSON)

Each plan stores its mapping definition as a JSONB column:

{ "sourceSchema": [ { "name": "member_name", "type": "string", "path": "member_name" }, { "name": "join_date", "type": "string", "path": "join_date" } ], "targetSchema": [ { "name": "fullName", "type": "string", "path": "contact.fullName" }, { "name": "memberSince", "type": "date", "path": "membership.startDate" } ], "mappings": [ { "id": "m1", "sourceField": "member_name", "targetField": "fullName", "transformation": "TRIM | TITLE_CASE" }, { "id": "m2", "sourceField": "join_date", "targetField": "memberSince", "transformation": "DATE_PARSE('MM/dd/yyyy')" } ], "validationRules": [ { "field": "fullName", "rule": "NOT_EMPTY", "message": "Name is required" }, { "field": "memberSince", "rule": "VALID_DATE", "message": "Invalid date format" } ] }

Execution Flow

1. User submits Plan for execution (Dry Run or Full Run) 2. PlanRun record created with status QUEUED 3. Server picks up run, sets status to RUNNING 4. For each record in source data: a. Apply field mappings b. Execute transformation DSL c. Validate against rules d. On Dry Run: record result but don't commit e. On Full Run: write to target database 5. Set status to COMPLETED or FAILED 6. Store result summary and errors as JSONB 7. Broadcast completion event via WebSocket

Logging Pipeline

Structured logging with automatic trace correlation ensures every log line can be linked back to a distributed trace.

Log Pattern (Console + File): %d{ISO8601} [%thread] %-5level %logger{36} traceId=%X{trace_id} spanId=%X{span_id} - %msg%n Profile-based routing: local → Console appender + Rolling file (100MB, 30-day retention, 3GB cap) docker/k8s → Console appender + OTEL appender (ships to Collector → Loki)

Log → Trace Correlation

Every log line includes the OpenTelemetry trace_id and span_id. Grafana's Loki datasource is configured with a derived field that turns trace IDs into clickable links to Tempo, enabling instant navigation from a log line to the full distributed trace.

Distributed Tracing

OpenTelemetry auto-instrumentation captures traces across HTTP requests, database calls, and Redis operations. Traces are exported via OTLP to the OTEL Collector, which forwards them to Tempo.

Trace → Metric Correlation

Tempo's metrics generator produces span metrics and service graphs, written back to Prometheus via remote write. This enables:

  • Service Map: Visual graph of service dependencies with request rates and error percentages
  • Span Metrics: Auto-generated latency histograms and throughput counters from trace data
  • Exemplars: Direct links from Prometheus metric data points to individual traces

Horizontal Scaling — Single Pane of Glass

All observability components are designed for multi-instance awareness:

  • Prometheus scrapes all API instances via Docker DNS or Kubernetes service discovery. Every metric includes an instance label.
  • Infrastructure dashboard shows all instances side-by-side: CPU, heap, request rate, thread count, HikariCP pools per instance.
  • Stat panels aggregate across instances using sum() and count(up) for total request rate, healthy instance count, and overall error rate.
  • Loki aggregates logs from all instances with the service_name label, queryable by instance.
  • Tempo captures distributed traces across instances, showing the full request path through load-balanced services.
  • Redis Pub/Sub ensures WebSocket events reach all instances, so the WebSocket connection gauge reflects the true global count.
Kubernetes-Ready: The Prometheus scrape config includes a commented Kubernetes service discovery block. When deployed to K8s, simply enable the kubernetes_sd_configs section — Prometheus will automatically discover and scrape all pods with the mxsuite-api label.

Audit Trail

  • Append-only audit_events table — never modified or deleted
  • Captures: actor, action, entity, before/after state (JSONB), trace ID, IP address
  • Distinguishes platform actions (GrowthZone staff) vs. tenant actions (customer)
  • Queryable by tenant, entity, actor, date range, platform-only filter
  • Trace ID in audit events links each audit record back to the full distributed trace in Tempo
  • Custom audit.events Micrometer counter tracks audit event rates on the Plan Execution dashboard

MXSuite is delivered as a set of Docker containers with a mounted .env file as the single source of configuration. All data persists outside the containers via bind-mounted host directories.

Bootstrap Sequence

On first startup, MXSuite automatically initializes itself:

  1. Flyway runs all migrations (V1 schema → V2 dev seed → V3 invitations/reset) and creates the database schema
  2. DataInitializer checks if a PLATFORM_ADMIN user exists. If not, creates one from MXSUITE_ADMIN_EMAIL and MXSUITE_ADMIN_PASSWORD in the .env file
  3. The system is immediately usable — log in as the platform admin and start creating tenants
