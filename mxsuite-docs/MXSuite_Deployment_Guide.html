2. Quick Start (Development)

Get a fully functional local environment running in under five minutes.

Step 1 — Clone the Repository

git clone https://github.com/your-org/mxsuite-api.git cd mxsuite-api

Step 2 — Create Your Local Environment File

cp .env.example .env

Open .env and review the defaults. For local development the defaults are sufficient, but you may want to set MXSUITE_ADMIN_EMAIL and MXSUITE_ADMIN_PASSWORD to your preferred credentials.

Step 3 — Start All Services

docker compose up -d

This single command starts PostgreSQL, Redis, the MXSuite API, the Shell front-end, the Admin panel, Prometheus, and Grafana. On first startup:

  1. Flyway runs all migration scripts and auto-creates the database schema.
  2. DataInitializer detects an empty database, creates the platform tenant, and provisions the platform admin user using the credentials from .env.
  3. Prometheus begins scraping metrics from /actuator/prometheus.
  4. Grafana provisions its four pre-built dashboards automatically.

Step 4 — Verify

ServiceURLNotes
MXSuite APIhttp://localhost:8080REST + WebSocket
Shell (Member Portal)http://localhost:3000React SPA
Admin Panelhttp://localhost:3001React SPA
Grafanahttp://localhost:3100Default login: admin / admin
Prometheushttp://localhost:9090Metrics query UI
Health Checkhttp://localhost:8080/actuator/healthShould return {"status":"UP"}
Success: If /actuator/health returns {"status":"UP"} and you can log in to the Shell at port 3000 with your admin credentials, your development environment is fully operational.

3. Production Deployment

Deployment Model

MXSuite is delivered as a set of Docker images. The customer (or ops team) receives pre-built container images and a docker-compose.yml file. The .env file is the single source of all configuration — it is mounted into the containers at runtime via a Docker Compose volume. No application code needs to be modified for production.

Production Host ├── /opt/mxsuite/ │ ├── docker-compose.yml <-- delivered with release │ ├── .env <-- customer-specific config │ ├── prometheus/ │ │ └── prometheus.yml <-- delivered with release │ └── grafana/ │ └── provisioning/ <-- delivered with release │ ├── /data/mxsuite/postgres/ <-- POSTGRES_DATA_PATH (bind mount) ├── /data/mxsuite/redis/ <-- REDIS_DATA_PATH (bind mount) └── /data/mxsuite/files/ <-- FILE_STORAGE_PATH (bind mount)

The .env File — Single Source of Configuration

Every runtime setting — database credentials, JWT secrets, SMTP configuration, storage paths, and feature flags — lives in a single .env file. This file is never baked into the Docker image. Instead, it is mounted at deploy time, making the same image deployable across staging, UAT, and production with zero rebuilds.

Warning: The .env file contains secrets (database passwords, JWT signing keys, SMTP credentials). Protect it with strict file permissions: chmod 600 .env and chown root:root .env.

Example Production .env

# ── Database ────────────────────────────────────────────── POSTGRES_HOST=mxsuite-db POSTGRES_PORT=5432 POSTGRES_DB=mxsuite POSTGRES_USER=mxsuite_prod POSTGRES_PASSWORD=<STRONG-RANDOM-PASSWORD-HERE> # ── Redis ───────────────────────────────────────────────── REDIS_HOST=mxsuite-redis REDIS_PORT=6379 # ── Auth / JWT ──────────────────────────────────────────── JWT_SECRET=<BASE64-ENCODED-256-BIT-SECRET> JWT_EXPIRATION_MS=3600000 JWT_REFRESH_EXPIRATION_MS=86400000 # ── Bootstrap Admin ─────────────────────────────────────── MXSUITE_ADMIN_EMAIL=admin@customer.org MXSUITE_ADMIN_PASSWORD=<INITIAL-ADMIN-PASSWORD> # ── Email / SMTP ────────────────────────────────────────── SMTP_HOST=smtp.customer.org SMTP_PORT=587 SMTP_USERNAME=mxsuite@customer.org SMTP_PASSWORD=<SMTP-PASSWORD> SMTP_FROM=noreply@customer.org SMTP_STARTTLS=true # ── Storage ─────────────────────────────────────────────── FILE_STORAGE_PATH=/data/files # ── Docker Volume Bind Mounts (host paths) ──────────────── POSTGRES_DATA_PATH=/data/mxsuite/postgres REDIS_DATA_PATH=/data/mxsuite/redis FILE_STORAGE_HOST_PATH=/data/mxsuite/files # ── Observability ───────────────────────────────────────── GRAFANA_ADMIN_PASSWORD=<GRAFANA-PASSWORD> PROMETHEUS_RETENTION=30d # ── Application ─────────────────────────────────────────── SPRING_PROFILES_ACTIVE=prod CORS_ALLOWED_ORIGINS=https://app.customer.org,https://admin.customer.org SERVER_PORT=8080

Starting the Platform

# Create data directories sudo mkdir -p /data/mxsuite/{postgres,redis,files} sudo chown -R 1000:1000 /data/mxsuite # Start all services docker compose --env-file /opt/mxsuite/.env up -d

The --env-file flag tells Docker Compose to read all variables from the specified file and inject them into the container environment and into variable interpolation within docker-compose.yml.

Named Volumes vs. Bind Mounts

StrategyUsed InHow It WorksPros / Cons
Named Volumes Development Docker manages volumes internally (e.g., mxsuite_pgdata) Simple setup; data lives inside Docker's storage directory; harder to back up directly
Bind Mounts Production Containers mount host directories (e.g., /data/mxsuite/postgres) Full control over storage location; easy to back up with standard tools; requires host directory setup

In the production docker-compose.yml, bind mounts are configured using the POSTGRES_DATA_PATH, REDIS_DATA_PATH, and FILE_STORAGE_HOST_PATH variables from .env:

services: mxsuite-db: image: postgres:16-alpine volumes: - ${POSTGRES_DATA_PATH}:/var/lib/postgresql/data mxsuite-redis: image: redis:7-alpine volumes: - ${REDIS_DATA_PATH}:/data mxsuite-api: image: mxsuite/api:latest volumes: - ${FILE_STORAGE_HOST_PATH}:/data/files

5. Database Management

Flyway Migration Strategy

MXSuite uses Flyway for versioned, repeatable database migrations. Migrations are embedded in the API container image and run automatically every time the application starts. This means the schema is always up to date — there is no separate "migration step" in the deployment process.

MigrationPurposeDetails
V1__initial_schema.sqlCore schemaAll tables, indexes, constraints, and enum types for the complete data model (tenants, users, members, events, invoices, etc.)
V2__dev_seed_data.sqlDevelopment seed dataSample tenants, users, and records for local development. Conditionally skipped in production via Flyway's ignoreMigrationPatterns or profile-specific config.
V3__invitations_and_reset.sqlInvitations & password resetTables for invitation tokens, password reset tokens, and email verification flows.

Schema Auto-Creation

On first startup against an empty database, Flyway detects that no flyway_schema_history table exists and runs all migrations from V1 onward. The schema is fully created without any manual SQL execution.

Hibernate Validation Mode

ddl-auto: validate — Hibernate is configured to validate the schema against JPA entity mappings at startup. It will never create, alter, or drop tables. If a mismatch is detected (e.g., a missing column), the application fails fast with a clear error. This ensures Flyway is the sole owner of schema changes.

Adding New Migrations

  1. Create a new file in src/main/resources/db/migration/ following the naming convention: V{N}__{description}.sql (two underscores after the version number).
  2. Write idempotent SQL where possible (use IF NOT EXISTS, IF EXISTS).
  3. Test locally by running docker compose down -v && docker compose up (the -v flag removes volumes, simulating a fresh database).
  4. Commit the migration file. It will be included in the next Docker image build.
Warning: Never modify a migration that has already been applied. Flyway checksums each migration file and will refuse to start if a checksum changes. Always create a new migration file for schema changes.

Backup Strategy

Full Database Backup

# Backup to compressed file docker exec mxsuite-db pg_dump -U mxsuite -d mxsuite \ --format=custom --compress=9 \ --file=/tmp/mxsuite_backup_$(date +%Y%m%d_%H%M%S).dump # Copy from container to host docker cp mxsuite-db:/tmp/mxsuite_backup_*.dump /backups/postgres/

Schema-Only Backup

docker exec mxsuite-db pg_dump -U mxsuite -d mxsuite \ --schema-only --file=/tmp/mxsuite_schema.sql

Automated Daily Backup (cron)

# Add to crontab: crontab -e 0 2 * * * /opt/mxsuite/scripts/backup.sh >> /var/log/mxsuite-backup.log 2>&1

Restore

# Stop the API to prevent writes during restore docker compose stop mxsuite-api # Restore from backup docker exec -i mxsuite-db pg_restore -U mxsuite -d mxsuite \ --clean --if-exists \ /tmp/mxsuite_backup_20260715_020000.dump # Restart the API docker compose start mxsuite-api

6. Bootstrap & First Login

How DataInitializer Works

On every application startup, the DataInitializer component checks whether the database contains any tenants. If the database is empty (first startup), it performs the following actions automatically:

  1. Creates the platform tenant — a special system tenant with the slug platform that represents the MXSuite platform itself.
  2. Creates the platform admin user using the email from MXSUITE_ADMIN_EMAIL and the password from MXSUITE_ADMIN_PASSWORD.
  3. Assigns the PLATFORM_ADMIN role to this user, granting full system access.
Warning: If MXSUITE_ADMIN_PASSWORD is not set in .env, the application will still start but will log a prominent warning:

WARN [DataInitializer] No MXSUITE_ADMIN_PASSWORD set. Platform admin created without a password. Set the variable and restart, or use the password reset flow.

Security: Change the Admin Password

The initial admin password is stored in plain text in the .env file. After first login, change the password immediately through the Admin Panel, then remove MXSUITE_ADMIN_PASSWORD from the .env file. The DataInitializer only runs on an empty database, so removing the variable after bootstrap has no effect on the existing admin account.

Step-by-Step: First Login to First User

StepActionDetails
1Log in as Platform AdminOpen https://admin.customer.org (or http://localhost:3001 in dev). Enter the admin email and password from .env.
2Change your passwordNavigate to Profile → Security and set a strong password. Remove MXSUITE_ADMIN_PASSWORD from .env.
3Create the first customer tenantGo to Platform Admin → Tenants → Create Tenant. Enter the organization name, slug, and contact email. The tenant is created immediately.
4Invite the first tenant adminSwitch context to the new tenant. Go to Users → Invite User. Enter the user's email and select the TENANT_ADMIN role. An invitation email is sent via SMTP.
5User accepts invitationThe invited user clicks the link in their email, sets their password, and logs in. They can now manage the tenant, invite more users, and configure modules.
Tip: The invitation email contains a time-limited token. If it expires, the tenant admin can resend the invitation from the Admin Panel.

7. SSL/TLS & Reverse Proxy

Architecture

In production, an nginx reverse proxy sits in front of the MXSuite services, handling SSL termination, HTTP/2, and request routing. The API and front-end containers only speak plain HTTP internally.

Internet │ ▼ ┌──────────────────────────────────┐ │ nginx (port 443 / SSL) │ │ ┌────────────────────────────┐ │ │ │ /api/* → mxsuite-api │ │ (port 8080) │ │ /ws/* → mxsuite-api │ │ (WebSocket upgrade) │ │ /admin/* → mxsuite-admin │ │ (port 3001) │ │ /* → mxsuite-shell │ │ (port 3000) │ └────────────────────────────┘ │ └──────────────────────────────────┘

Example nginx Configuration

upstream mxsuite_api { server mxsuite-api:8080; # Uncomment for horizontal scaling: # server mxsuite-api-2:8080; } server { listen 80; server_name app.customer.org admin.customer.org; return 301 https://$host$request_uri; } server { listen 443 ssl http2; server_name app.customer.org; ssl_certificate /etc/nginx/ssl/fullchain.pem; ssl_certificate_key /etc/nginx/ssl/privkey.pem; ssl_protocols TLSv1.2 TLSv1.3; ssl_ciphers HIGH:!aNULL:!MD5; # ── Proxy headers ────────────────────────────── proxy_set_header Host $host; proxy_set_header X-Real-IP $remote_addr; proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for; proxy_set_header X-Forwarded-Proto $scheme; # ── API ──────────────────────────────────────── location /api/ { proxy_pass http://mxsuite_api; } # ── WebSocket ────────────────────────────────── location /ws/ { proxy_pass http://mxsuite_api; proxy_http_version 1.1; proxy_set_header Upgrade $http_upgrade; proxy_set_header Connection "upgrade"; proxy_read_timeout 86400s; } # ── Actuator (restrict to internal) ──────────── location /actuator/ { deny all; } # ── Shell front-end ──────────────────────────── location / { proxy_pass http://mxsuite-shell:3000; } } server { listen 443 ssl http2; server_name admin.customer.org; ssl_certificate /etc/nginx/ssl/fullchain.pem; ssl_certificate_key /etc/nginx/ssl/privkey.pem; ssl_protocols TLSv1.2 TLSv1.3; ssl_ciphers HIGH:!aNULL:!MD5; proxy_set_header Host $host; proxy_set_header X-Real-IP $remote_addr; proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for; proxy_set_header X-Forwarded-Proto $scheme; location /api/ { proxy_pass http://mxsuite_api; } location /ws/ { proxy_pass http://mxsuite_api; proxy_http_version 1.1; proxy_set_header Upgrade $http_upgrade; proxy_set_header Connection "upgrade"; proxy_read_timeout 86400s; } location / { proxy_pass http://mxsuite-admin:3001; } }

Key Proxy Headers

HeaderPurpose
X-Forwarded-ForPasses the client's real IP address to the API for audit logging and rate limiting.
X-Forwarded-ProtoTells Spring Boot that the original request was HTTPS, ensuring generated URLs use the correct scheme.
X-Real-IPAlternative single-value client IP header, used by some logging frameworks.
Upgrade / ConnectionRequired for WebSocket connections to be properly upgraded through the proxy.
Let's Encrypt: For automated SSL certificate management, use Certbot with the nginx plugin: certbot --nginx -d app.customer.org -d admin.customer.org

8. Horizontal Scaling

Overview

MXSuite is designed to scale horizontally. The API service is stateless (session state lives in Redis, file storage is on a shared mount), so you can run multiple instances behind a load balancer with no application changes.

Adding a Second API Instance

The docker-compose.yml includes a pre-configured but commented-out second API instance. To enable it:

# In docker-compose.yml, uncomment the mxsuite-api-2 service: mxsuite-api-2: image: mxsuite/api:latest env_file: .env environment: - SERVER_PORT=8080 depends_on: mxsuite-db: condition: service_healthy mxsuite-redis: condition: service_healthy volumes: - ${FILE_STORAGE_HOST_PATH}:/data/files healthcheck: test: ["CMD", "curl", "-f", "http://localhost:8080/actuator/health"] interval: 30s timeout: 10s retries: 3

Then update the nginx upstream to include both instances:

upstream mxsuite_api { server mxsuite-api:8080; server mxsuite-api-2:8080; }

Restart the stack:

docker compose --env-file /opt/mxsuite/.env up -d

Redis Pub/Sub for WebSocket Events

When a user performs an action (e.g., creating a member record), the API publishes a WebSocket event. In a multi-instance deployment, the event must reach all connected clients, regardless of which API instance they are connected to. MXSuite uses Redis Pub/Sub as the broadcast channel:

┌───────────────────┐ Redis Pub/Sub ┌───────────────────┐ │ mxsuite-api-1 │ ◄──────────────────── │ mxsuite-api-2 │ │ (WebSocket users │ ────────────────────► │ (WebSocket users │ │ connected here) │ │ connected here) │ └───────────────────┘ └───────────────────┘ ▲ ▲ │ │ ┌─────────┐ ┌─────────┐ │ Client A│ │ Client B│ └─────────┘ └─────────┘

This is fully automatic. No configuration is needed beyond ensuring all API instances connect to the same Redis instance.

Prometheus Auto-Discovery

The Prometheus configuration uses Docker service discovery to automatically find all MXSuite API instances. When you add mxsuite-api-2, Prometheus detects it and begins scraping metrics from both instances without any configuration changes.

Grafana Infrastructure Dashboard

The pre-provisioned Infrastructure dashboard in Grafana is designed for multi-instance deployments. It includes a template variable for instance, allowing you to view metrics for individual instances or all instances side by side. Panels include per-instance CPU, memory, request rates, and error counts.

The MXSuite API includes an EmailService.sendAdminAlert() method that sends critical alerts directly via the configured SMTP server. This is used for application-level alerts (e.g., failed payment processing, tenant provisioning errors) that are detected in Java code rather than through metrics.

Configuring Grafana SMTP Alerting

To have Grafana send alert notifications via email, configure SMTP in the Grafana container environment:

# Add to docker-compose.yml under the grafana service: environment: - GF_SMTP_ENABLED=true - GF_SMTP_HOST=${SMTP_HOST}:${SMTP_PORT} - GF_SMTP_USER=${SMTP_USERNAME} - GF_SMTP_PASSWORD=${SMTP_PASSWORD} - GF_SMTP_FROM_ADDRESS=grafana@customer.org

Then in the Grafana UI, create a Contact Point of type "Email" and add the recipient addresses. Attach this contact point to your alert rules.

10. Backup & Recovery

Backup Components

A complete MXSuite backup consists of three components:

ComponentDataMethodFrequency
PostgreSQLAll application data (tenants, users, members, events, invoices, etc.)pg_dumpDaily + before upgrades
File StorageUploaded documents, logos, attachmentsrsync or cpDaily
RedisSession cache, WebSocket state (ephemeral, but convenient to preserve)RDB snapshots (automatic)Automatic (appendonly yes)

PostgreSQL Backup

# Full compressed backup docker exec mxsuite-db pg_dump \ -U ${POSTGRES_USER} \ -d ${POSTGRES_DB} \ --format=custom \ --compress=9 \ -f /tmp/mxsuite_$(date +%Y%m%d_%H%M%S).dump # Copy to backup location docker cp mxsuite-db:/tmp/mxsuite_*.dump /backups/postgres/ # Verify backup integrity docker exec mxsuite-db pg_restore \ --list /tmp/mxsuite_*.dump > /dev/null && echo "Backup valid"

File Storage Backup

# Sync file storage to backup location (incremental) rsync -avz --delete \ /data/mxsuite/files/ \ /backups/files/ # Or simple copy cp -a /data/mxsuite/files/ /backups/files-$(date +%Y%m%d)/

Redis Backup

Redis is configured with appendonly yes in the Docker Compose setup, meaning every write is persisted to the append-only file (AOF). Redis also takes periodic RDB snapshots. Both files are stored in the REDIS_DATA_PATH bind mount.

# Copy Redis data directory cp -a /data/mxsuite/redis/ /backups/redis-$(date +%Y%m%d)/

Full Recovery Procedure

Follow these steps to recover MXSuite from a complete backup:

  1. Stop the stack
    docker compose --env-file /opt/mxsuite/.env down
  2. Restore PostgreSQL data
    # Start only the database docker compose --env-file /opt/mxsuite/.env up -d mxsuite-db # Wait for PostgreSQL to be ready docker exec mxsuite-db pg_isready -U mxsuite # Drop and recreate the database docker exec mxsuite-db psql -U mxsuite -c "DROP DATABASE IF EXISTS mxsuite;" docker exec mxsuite-db psql -U mxsuite -c "CREATE DATABASE mxsuite OWNER mxsuite;" # Restore from backup docker cp /backups/postgres/mxsuite_20260715_020000.dump mxsuite-db:/tmp/ docker exec mxsuite-db pg_restore \ -U mxsuite -d mxsuite \ --clean --if-exists \ /tmp/mxsuite_20260715_020000.dump
  3. Restore file storage
    rsync -avz /backups/files/ /data/mxsuite/files/
  4. Restore Redis (optional)
    cp -a /backups/redis-20260715/ /data/mxsuite/redis/
  5. Start the full stack
    docker compose --env-file /opt/mxsuite/.env up -d
  6. Verify recovery
    # Check health curl -s http://localhost:8080/actuator/health | jq . # Check database connectivity docker exec mxsuite-db psql -U mxsuite -d mxsuite \ -c "SELECT count(*) FROM tenants;" # Check application logs for errors docker logs --tail 50 mxsuite-api
Recovery verified: If /actuator/health returns {"status":"UP"} and the tenant count matches your expected data, the recovery is complete.

11. Troubleshooting

Common Issues

Port Conflicts

Symptom: Error: Bind for 0.0.0.0:5432 failed: port is already allocated

Cause: Another process (often a locally installed PostgreSQL) is already using the port.

Fix:

# Find what's using the port sudo lsof -i :5432 # Either stop the conflicting service or change the port in .env # For example, map to a different host port in docker-compose.yml: # ports: # - "15432:5432"

Flyway Migration Errors

Symptom: Application fails to start with FlywayException: Validate failed: Migration checksum mismatch

Cause: A previously applied migration file was modified.

Fix: Never modify applied migrations. If you must fix a mistake, create a new migration. In development, you can reset the database:

# DEVELOPMENT ONLY — destroys all data docker compose down -v docker compose up -d

SMTP Connection Failed

Symptom: MailSendException: Could not connect to SMTP host

Cause: Incorrect SMTP settings, firewall blocking port 587, or missing credentials.

Fix:

# Verify SMTP connectivity from inside the container docker exec mxsuite-api nc -zv ${SMTP_HOST} ${SMTP_PORT} # Check .env values grep SMTP .env # Common fixes: # - Ensure SMTP_STARTTLS=true for port 587 # - Some providers require app-specific passwords # - Check firewall rules: sudo ufw allow out 587/tcp

JWT Token Expired / Authentication Errors

Symptom: API returns 401 Unauthorized with message JWT token is expired

Cause: The access token has expired (default: 1 hour).

Fix: The front-end should automatically refresh tokens using the refresh token endpoint. If the refresh token has also expired (default: 24 hours), the user must log in again. If all users are affected simultaneously, check that JWT_SECRET has not changed between deployments.

Container Fails Health Check

Symptom: docker ps shows (unhealthy) status for mxsuite-api.

Fix:

# Check application logs docker logs --tail 100 mxsuite-api # Manually test health endpoint docker exec mxsuite-api curl -s http://localhost:8080/actuator/health # Common causes: # - Database not yet ready (check mxsuite-db health) # - Out of memory (check docker stats) # - Flyway migration failure (check logs for FlywayException)

Health Check Endpoint

The MXSuite API exposes a comprehensive health check at /actuator/health:

curl -s http://localhost:8080/actuator/health | jq . # Response: { "status": "UP", "components": { "db": { "status": "UP", "details": { "database": "PostgreSQL" } }, "redis": { "status": "UP" }, "diskSpace": { "status": "UP", "details": { "free": "85 GB" } }, "mail": { "status": "UP" } } }

If any component reports DOWN, the overall status becomes DOWN and the HTTP response code changes to 503 Service Unavailable.

12. Security Checklist for Production

Complete every item on this checklist before exposing MXSuite to the internet.

#ActionHowRisk if Skipped
1 Change JWT_SECRET Generate a cryptographically random 256+ bit key:
openssl rand -base64 32
Set in .env as JWT_SECRET. 		Attacker can forge authentication tokens and impersonate any user.
2 Change POSTGRES_PASSWORD Generate a strong random password and set in .env. Restart the database and API. Database compromise if an attacker gains network access to port 5432.
3 Change Grafana admin password Set GRAFANA_ADMIN_PASSWORD in .env or change via Grafana UI on first login. Unauthorized access to infrastructure metrics and alert configuration.
4 Set proper CORS origins In .env, set CORS_ALLOWED_ORIGINS to your production domains only. Remove localhost entries. Cross-origin attacks from unauthorized domains.
5 Enable SSL/TLS Configure nginx with a valid SSL certificate (see Section 7). Redirect all HTTP to HTTPS. Credentials and data transmitted in plain text; susceptible to man-in-the-middle attacks.
6 Set and then remove MXSUITE_ADMIN_PASSWORD Set MXSUITE_ADMIN_PASSWORD in .env for first startup. After first login and password change, remove the variable from .env. Admin password stored in plain text on disk indefinitely.
7 Configure firewall rules
## Only expose nginx (443) to the internet sudo ufw default deny incoming sudo ufw allow 22/tcp # SSH sudo ufw allow 443/tcp # HTTPS (nginx) sudo ufw enable ## Block direct access to internal services: ## 8080 (API) — only reachable via nginx ## 5432 (Postgres) — only reachable within Docker network ## 6379 (Redis) — only reachable within Docker network ## 9090 (Prometheus) — only reachable within Docker network ## 3100 (Grafana) — access via VPN or SSH tunnel
 Internal services (database, Redis, Prometheus) directly accessible from the internet.
Critical: Items 1 and 2 (JWT_SECRET and POSTGRES_PASSWORD) are the most important. The default development values are well-known and provide zero security. Never deploy to production without changing them.

Additional Hardening

  • Docker socket: Do not expose the Docker socket (/var/run/docker.sock) to any container. None of the MXSuite services require it.
  • Non-root containers: The MXSuite API image runs as a non-root user by default. Do not override this with user: root.
  • Network segmentation: The Docker Compose file defines internal networks. Only nginx should be on the external network; all other services communicate on the internal network.
  • Log retention: Configure Docker log rotation to prevent disk exhaustion:
# /etc/docker/daemon.json { "log-driver": "json-file", "log-opts": { "max-size": "50m", "max-file": "5" } }
  • Regular updates: Keep Docker, the host OS, and the base images updated. Subscribe to security advisories for PostgreSQL, Redis, and Java.
  • Secrets management: For environments with strict compliance requirements, consider using Docker Secrets or HashiCorp Vault instead of .env files.
Production-ready: Once every item on this checklist is complete, your MXSuite deployment meets baseline security standards for a production SaaS platform.
